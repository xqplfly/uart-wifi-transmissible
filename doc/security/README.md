# 安全设计

本文档说明工程当前的安全加固实现，重点关注“输入边界收口”“异常处理”和“敏感信息最小暴露”。

## 1. 设计目标

当前安全层主要解决以下问题：

- 防止串口、Web、TCP 输入导致缓冲区失控
- 防止非法字符、危险命令或恶意帧直接进入透传链路
- 对连续错误输入做速率抑制与来源封禁
- 避免在日志、串口调试和 Web 页面中泄露敏感网络信息

非目标：

- 不提供 TLS
- 不提供身份认证或密钥交换
- 不提供消息签名或防重放协议

## 2. 受保护的输入源

安全层为不同入口维护独立状态：

- USB CDC
- Web 输入
- TCP 客户端入站
- TCP 服务器各客户端入站

这样可以避免某一路异常流量影响其他来源。

## 3. 安全帧格式

所有外部透传输入统一使用以下帧格式：

```text
@<payload_length>:<payload>#
```

示例：

```text
@12:DATA:HELLO\r\n#
```

解析时会校验：

1. 帧总长度是否合法
2. 帧头是否为 @
3. 帧尾是否为 #
4. 长度字段是否全为数字
5. 负载长度是否与长度字段一致

## 4. 关键阈值

| 常量 | 当前值 | 说明 |
| --- | --- | --- |
| SECURITY_MAX_TRANSPARENT_PAYLOAD | 240 | 单次允许的最大负载长度 |
| SECURITY_MAX_FRAME_LENGTH | 320 | 单次允许的最大帧长度 |
| SECURITY_MAX_INVALID_ATTEMPTS | 5 | 连续错误次数上限 |
| SECURITY_INPUT_TIMEOUT_MS | 2000 ms | 半包超时清空阈值 |
| SECURITY_LOCKOUT_MS | 30000 ms | 达到错误上限后的封禁时长 |

## 5. 校验链路

### 5.1 字节级校验

- 帧缓冲累计长度不能超过上限
- 帧中只能出现允许的可打印字符、回车、换行和制表符
- 发现非法字节时立即清空当前缓冲并记录失败

### 5.2 负载级校验

- 负载不能为空
- 负载长度不能超过 240 字节
- 负载中不能包含帧头/帧尾保留字符

### 5.3 白名单校验

当前允许的主要前缀包括：

- AT+
- CLIENT_ID:
- SERVER:
- CMD:
- DATA:
- REQ:
- RESP:
- STATUS
- PING / PONG
- READ / WRITE / SET / GET / INFO / QUERY / MEAS
- 以 $, ! 开头的遥测类数据

此外，短的状态型负载也允许以下精确匹配：

- OK
- ACK / NACK
- ON / OFF
- TRUE / FALSE
- 0 / 1

对未命中显式白名单的负载，会尝试按“遥测风格文本”规则匹配；如果仍不符合，则拒绝透传。

### 5.4 黑名单拦截

当前显式拦截的危险片段包括：

- AT+RST
- AT+RESET
- AT+RESTART
- AT+RESTORE
- AT+CONFIG
- AT+EXITCONFIG
- AT+CIPSTART
- AT+CIPSERVER
- AT+CWMODE
- +++
- ../ 和 ..\\
- <SCRIPT

说明：白名单与黑名单同时存在，黑名单优先用于快速阻断典型危险输入，白名单用于限制总体输入面。

## 6. 错误处理与攻击缓解

### 6.1 连续失败计数

每个输入源独立维护 failureCount。

- 校验失败时递增
- 校验成功时清零
- 达到阈值后设置 blockUntil

### 6.2 来源封禁

当 failureCount 达到 5 次：

- 当前输入源会被封禁 30 秒
- 新输入会直接拒绝
- 到期后自动恢复

### 6.3 半包超时

如果帧已开始接收但在 2 秒内没有形成完整数据：

- 清空当前来源的帧缓冲
- 记录一次失败

这个机制用于限制半包堆积、异常粘包和慢速消耗类输入。

## 7. 敏感信息最小化暴露

当前已经收口以下敏感信息输出：

- Web 配置页中的 SSID 和密码不回显原文
- 串口调试信息不再明文打印 WiFi 密码
- AP SSID、客户端 SSID 和 IP 地址在 Web/串口/日志中采用脱敏形式
- 服务器模式系统日志记录客户端地址时只保留前两段网段

## 8. 与其他模块的协作

| 模块 | 协作方式 |
| --- | --- |
| uart_utility.ino | USB 入口统一走 appendIngressChunk()/getValidatedPayload() |
| client_server_mode.ino | TCP Client/Server 入站统一走安全帧解析 |
| web_server.ino | Web 输入先组帧，再做统一校验 |
| uart_interrupt.ino | UART2 出站切片后重新封装为安全帧 |

## 9. 当前局限

- 安全帧只校验结构和内容合法性，不校验消息来源身份。
- TCP 传输仍是明文链路，适合局域网和受控环境。
- 白名单采用规则匹配，适合当前场景，但无法替代业务级协议语义校验。

## 10. 后续可增强方向

1. 为客户端和服务器加入设备级鉴权口令
2. 对帧增加 CRC 或 MAC
3. 将 server_ip 改为可配置并增加地址合法性校验
4. 引入 TLS 或在上层网关增加安全隧道
5. 对 Web 管理页增加登录保护
